Why we're doing this

Docs are the first thing to fall behind. Here is what that costs us, and what we want instead.

What happens today

  • Docs drift silently, and by different amounts. Sampled repos range from 0 to 65 days behind their last code commit.
  • Markdown uses a mix of URLs and relative paths. Neither points at a single shared docs domain today, because that domain is not yet live. DEV-1079 DEV-1086
  • Command blocks in infra docs rarely name their target environment, even when the surrounding prose does. Commands written for EC2 can rot at the EKS cutover.
  • We don't yet have a shared answer for when docs are required versus optional.

What we want instead

  • Docs ship in the same PR as the code that changed them.
  • Cross service references resolve through a single shared docs domain, so they survive repo moves.
  • Infra command blocks name their environment explicitly.
  • A written policy tells the author, the reviewer, and the reader the same thing.
  • Existing habits stay in place. Most active repos already have a docs/ folder and a root CLAUDE.md. The policy formalizes what's already common.

The Policy

Six sections. Governance on top of the pipeline. DEV-1070

1

When docs are required

User-facing features, breaking API changes, new services, config changes across services. Not bug fixes, refactors, or dep bumps.

2

Who is responsible

Claude drafts on PR open. Author verifies, reviewer verifies in the same pass. No separate doc reviewer.

3

Where docs live

docs/users/ public, docs/dev/ private, docs/internal/ excluded. CLAUDE.md at root.

4

What docs contain

URLs for cross-service refs. Env tags on infra commands.

5

Enforcement

PR checkbox: docs updated, or N/A with reason. No CI gate in v1.

6

Publishing

GitBook auto-syncs from main. internal/ stays in the repo.
