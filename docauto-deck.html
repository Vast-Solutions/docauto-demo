Why we're doing this

Docs are the first thing to fall behind. Here is what that costs us, and what we want instead.

What happens today

  • Docs drift silently, and by different amounts. Sampled repos range from 0 to 65 days behind their last code commit.
  • Markdown uses a mix of URLs and relative paths. Neither points at a single shared docs domain today. DEV-1086
  • Not all command blocks in infra docs name their target environment. Commands written for EC2 can rot at the EKS cutover, for example.
  • No global rule for when docs are required, who writes them, or where they live.

What we want instead

  • Docs ship as a parallel commit alongside code, keyed by the same Linear issue ID.
  • One central knowledge base. Cross-service references resolve through a single shared docs domain, so they survive repo moves.
  • Command blocks name their environment explicitly.
  • Shared written rules for when docs are required, who writes them, and where they live.
  • One slash command at commit time wraps the doc decision into the developer's existing flow.

Documentation Policy

§1. When docs are required

Required

  • New user-facing feature or capability
  • Breaking change to a public API or interface
  • New service or infrastructure component
  • Config change that affects other services

Not required

  • Bug fixes with no user-visible behavior change
  • Refactors that preserve existing behavior
  • Dependency bumps, internal tooling, CI-only changes

§2. Who is responsible

The developer runs /commit at commit time. Claude reviews the working diff against the developer's description, decides whether documentation impact exists, and proposes documentation updates for approval before committing.

The same code reviewer approves both PRs together. No separate doc reviewer.

When documentation impact exists, two parallel commits are produced, keyed off the same Linear issue ID:

  1. Code commit on the service repo branch (e.g. feat/SCO-142-add-oauth-login)
  2. Documentation commit on a parallel branch in Visionary-Solutions/documentation (e.g. docs/SCO-142-add-oauth-login)

If a change has no documentation impact, /commit skips the doc branch. The reviewer verifies the absence is honest in the same pass as the code review.

§3. Where docs live

Single layer. Visionary-Solutions/documentation is the documentation source of truth. Code repos no longer carry their own docs/.

documentation/
├── user/                  # outside-facing, public GitBook space
│   └── <Product>/         # e.g. Scout/, VisionProcessor/
└── dev/                   # on-team, private GitBook space
    └── <Product>/

Pick the folder by audience. Use user/ for external consumers, sales, or stakeholders. Use dev/ for internal developers across teams.

Each product gets its own subfolder under each audience. Cross-cutting documentation that isn't owned by a single product (architecture, decisions, cross-service interfaces) lives at the audience level, for example dev/Architecture/ or dev/Decisions/.

§4. What docs contain

  1. Cross-service links use URLs. When Service A's doc refers to Service B, link to docs.visionaryauto.ai/<service>/..., not relative file paths. URLs survive repo moves and environment transitions; paths do not.
  2. Environment tags on infra details. Any deployment, config, or ops detail names its target explicitly ("on EC2," "post-EKS migration"). Undated commands rot at cutover.

§5. Enforcement

/commit is the gate. The developer cannot complete the commit step without making an explicit decision on documentation impact. Skipping /commit (committing manually with git commit) is a policy violation; the reviewer flags PRs that didn't go through it.

When documentation impact exists, both PRs (code repo + documentation/) are approved as a unit. No CI gate in v2.

§6. Publishing

GitBook auto-syncs from Visionary-Solutions/documentation to docs.visionaryauto.ai.

  • Public user/ at the root
  • Access-controlled dev/ under a private space

Walkthrough

From /start-task to /commit, end to end. DEV-1075

  1. Developer runs /start-task (or /pull-task). Claude creates a Linear issue, generates up to five subtasks, transitions the parent to In Progress, and cuts the scout branch.
  2. Developer writes code through the subtasks. Session ends with the working tree dirty, no commit.
  3. Developer opens a fresh session and runs /commit. Claude reads the diff, parses the Linear issue ID from the branch name, and reconciles the diff against the developer's description.
  4. Claude decides whether documentation impact exists. If it does, Claude proposes the doc branch name, the doc edits in Visionary-Solutions/documentation, and the doc commit message alongside the scout commit message in a single approval block.
  5. On approval, commits land in both repos. The Linear issue transitions to In Review.
  6. Developer pushes both repos manually and opens PRs in each. The same code reviewer approves both PRs together.
  7. On merge, GitBook syncs documentation/ and the docs publish to docs.visionaryauto.ai.

Documentation hygiene DEV-1104 DEV-1105

Three cleanups before the domain swap:

  • Rename one of the two GitBook "Docs" spaces
  • Audit remaining GitBook sync sources
  • Mark the legacy wiki-content repo as archive-only

Removes the duplicate-space confusion in GitBook and signals the canonical-direction shift to anyone reading along. Brings GitBook to a clean state for the domain cutover.

Slash commands DEV-1106

Build /start-task, /pull-task, and /commit as .claude/commands/ in scout and any code repo onboarding the policy.

Gates the v2 pilot. Without the commands shipped, there's no enforcement gate, only the policy text. Each repo opts in by installing the commands.

Domain cutover DEV-1086

Point visionarywiki.gitbook.io at the GitBook vs-users site. A DNS CNAME swap, no content moves.

Once it lands, the cross-service URL convention from §4 resolves to a real public domain. Every published doc gets stable external links instead of placeholders.
