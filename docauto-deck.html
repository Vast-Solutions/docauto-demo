Why we're doing this

Docs is the first automation we're shipping into the developer's flow. Onboarding, deployment, and the developer platform are next.

What happens today

  • Docs drift because they live separately from the commits that change behavior.
  • Cross-service references break when repos move.
  • No shared rule for when docs are required.

What we want instead

  • Docs ship in the same commit gesture as the code that demands them.
  • One central source of truth, one shared docs domain.
  • The same in-flow pattern extends to Onboarding, deployment, and the developer platform.

Documentation Policy

v3 closes gaps and minimizes ambiguity.

Why it works as a governing document

  • Decisive. Required, Optional, and Not Required are explicit. The classifier is named, not implicit.
  • Centralized. All docs live in Visionary-Solutions/documentation. No per-repo docs/ to drift.
  • Low-friction. The doc decision rides on the commit gesture. No new gate, no separate doc workflow to maintain.
  • Audience-driven. Each Required and Optional item names its destination folder inline (user/ vs dev/ subfolder).
  • Honest. Every gap is named. The Deliberate out-of-scope section says what the policy doesn't solve; nothing is hidden.
DEV-1128 · v3 (highlights)

Documentation Policy

§1. When docs are required

Required → user/<Product>/: new user-facing feature or capability; breaking change to an externally-consumed API.

Required → dev/<Product>/ (or dev/Architecture/ if cross-cutting): breaking change to an internally-consumed cross-service API; new service or infrastructure component; config change that affects other services.

Optional (/commit asks; default no): ADRs → dev/Decisions/; internal postmortems → dev/Operations/; archaeology and tribal knowledge → dev/<Product>/.

Not required: bug fixes with no user-visible change, behavior-preserving refactors, dep bumps, CI-only.

§2. Who is responsible

The developer runs /commit. It reads the diff and the developer's description and classifies against §1: mechanical patterns hard-skip as Not Required; description keywords (decision, rationale, postmortem) bias toward Optional; everything else is Required (and Required targets the §1-mapped folder).

Required (or opted-in Optional) produces two commits keyed off the same Linear ID: code on feat/SCO-9999-..., docs on docs/SCO-9999-... in Visionary-Solutions/documentation. Same reviewer approves both as a unit.

§3. Where docs live

Visionary-Solutions/documentation is the source of truth. Code repos no longer carry their own docs/.

documentation/
├── user/<Product>/                  # public, per-product subdomain
└── dev/<Product>/                   # internal, visitor-auth at
    dev/Architecture/, dev/Decisions/, # docs.visionaryauto.ai
    dev/Operations/

§4. What docs contain

  1. Cross-service links use URLs, not relative paths. Internal → docs.visionaryauto.ai/<service>/...; public → product docs subdomain (e.g. docs.scoutautos.com/...).
  2. Environment tags on infra details. Name the target explicitly ("on EC2," "post-EKS migration").

§5. Enforcement

/commit is the gate for Required. Skipping it is a policy violation (except §7 hotfix). Both PRs are approved as a unit. No CI gate.

§6. Publishing

GitBook auto-syncs. dev/ → docs.visionaryauto.ai via vs-docs (visitor-auth). user/<Product>/ → product docs subdomain (e.g. docs.scoutautos.com).

§7. Hotfix exception

Bypassing /commit is allowed only for active production incidents (customer-facing impact). Authorization required; doc-debt follow-up issue opened, runs through /commit within 7 days.

§8. Measurement

Lagging indicators: reviewer comments asking about docs; open doc-debt count. Sustained two-quarter upward trend triggers a policy revision.

What's next

Three decisions pace the fleet rollout. None block the demo pilot.

Skills distribution DEV-1118

Skills ship in scout today. Phase 1 (interim install path in documentation/dev/Commands/, DEV-1119) shipped. Phase 1b graduates the canonical files to a dedicated Visionary-Solutions/claude-commands repo with v0.1 tag and catalog-info.yaml DEV-1127. Migration path leads to IDP-distributed golden paths once the platform tile lands.

Pilot activation DEV-1075

With v3 ratified, the next gate is proving the full workflow end to end on a real Linear-tracked task. A single developer runs /start-task, codes through subtasks, then /commit in a fresh session. Both PRs land together; GitBook auto-syncs on merge.

Scope is narrow on purpose: one developer, one change, full v3 flow. The readout informs broader onboarding across other code repos.

Publish as Claude Code plugin DEV-1120

After DEV-1127 lands, publish the claude-commands repo as a Claude Code plugin. Add .claude-plugin/plugin.json and a team marketplace; pre-enable auto-update so consumers stay current at session start with no manual command.

Commands become namespaced (/visionary:start-task). Same primitive, broader reach: any new repo installs the plugin and is in the docauto loop.
